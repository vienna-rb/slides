Different Techniques

Vertically

|   Database   |     Slave     |
+--------------+---------------+
| App Server 1 | App Server 2  |
+--------------+---------------+
|       Load Balancer(s)       |

Great for production. Doesn't help in Tests. Good for scaling, so don't dismiss it.

Horizontally

|        ORM         |
|        API         | < Use Workers for 'invisible' Tasks
|        DAO         |
|  Business Objects  | <
|     Presenters     | < Via Adapters
|  View  Generators  | <

Good for testing. Doesn't matter in Production. Makes changes even easier.

Why not use mock_model?

Consider: you are including all of this:

 1 #...
 2 stubs = stubs.reverse_merge(:id => next_id)
 3 stubs = stubs.reverse_merge(:persisted? => !!stubs[:id],
 4                             :destroyed? => false,
 5                             :marked_for_destruction? => false,
 6                             :valid? => true,
 7                             :blank? => false)
 8 
 9 double("#{model_class.name}_#{stubs[:id]}", stubs).tap do |m|
10   m.singleton_class.class_eval do
11   include ActiveModelInstanceMethods
12   include ActiveRecordInstanceMethods if defined?(ActiveRecord)
13   include ActiveModel::Conversion
14   include ActiveModel::Validations
15 end
16 if defined?(ActiveRecord)
17   [:save, :update_attributes, :update].each do |key|
18     if stubs[key] == false
19       m.errors.stub(:empty? => false)
20     end
21   end
22 end
23 m.__send__(:__mock_proxy).instance_eval(<<-CODE, __FILE__, __LINE__)
24 #...
25 #...

It's not bad per se, but if you just use state of the models, don't use it

Solution 1: Use an ad-hoc class

 1 class MyTestDummyModel < OpenStruct
 2   extend ActiveModel::Naming
 3 end
 4 
 5 describe TestController do
 6   context "GET /show" do
 7     let (:object) { MyTestDummyModel.new(id: index) }
 8     it "should work" do
 9       ObjectModel.stub!(:find).with("5").and_return(object)
10       get :index, id: 5
11       response.should be_success
12     end
13   end
14 end

Solution 2: Dependency Injection

A. Controller

 1 class TestController < ActionController::Base
 2   attr_accessor :single_fetch_function
 3   def initialize
 4     super
 5     @single_fetch_function ||= ObjectModel.method(:find)
 6   end
 7 
 8   def show
 9     @object =self.single_fetch_function.(params[:id])
10   end
11 
12 end

B. Test

 1 describe TestController do
 2   context "GET /show" do
 3     it "should work" do
 4       controller.single_fetch_function =
 5         ->(x) { MyTestDummyModel.new(id: x) }
 6       get :index, id: 5
 7       response.should be_success
 8     end
 9   end
10 end

You can use a mock here, but only if you need the interaction
